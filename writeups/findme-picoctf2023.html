Phase 00

Présentation du Challenge

findme est un challenge Web Exploitation de picoCTF 2023, créé par Geoffrey Njogu, noté 100 points (difficulté moyenne). L'énoncé donne une instruction précise :

"Help us test the form by submitting the username as test and password as test!"
challenge_card.png — picoCTF
findme — Web Exploitation, Medium, 100 pts, picoCTF 2023 — Indice : "any redirections?"

L'indice fourni — "any redirections?" — est la clé de tout le challenge : il oriente immédiatement vers l'étude du comportement de redirection HTTP du formulaire, plutôt que vers une faille classique d'authentification (injection, bypass de mot de passe...).

Phase 01

Fondamentaux Théoriques

Avant l'exploitation, il est indispensable de comprendre précisément les mécanismes HTTP et les outils mobilisés dans ce write-up.

Les redirections HTTP

Redirection HTTP (3xx)
Réponse qu'un serveur web envoie pour indiquer au client (navigateur, curl...) qu'il doit effectuer une nouvelle requête vers une autre URL. Le code 302 Found signale une redirection temporaire. L'URL de destination est précisée dans l'en-tête HTTP Location. Par défaut, les navigateurs suivent ces redirections automatiquement et de façon invisible pour l'utilisateur — c'est précisément ce comportement qui cache la vulnérabilité de ce challenge.
Header HTTP
Métadonnée transmise avec une requête ou une réponse HTTP (ex. Content-Type, Location, Cookie) — invisible dans le rendu visuel normal d'une page.
Location (header)
En-tête spécifique aux réponses de redirection, indiquant au client la nouvelle URL à charger.
Paramètre de requête
Donnée transmise dans l'URL après un "?" ou intégrée au chemin, ici id dans /next-page/id=....

L'encodage Base64

Base64
Système d'encodage qui convertit des données binaires ou textuelles en une chaîne de caractères ASCII imprimables (lettres, chiffres, +, /, complétée par = en fin de chaîne). Ce n'est pas un chiffrement : aucune clé secrète n'est requise pour décoder — l'opération est immédiate et réversible par quiconque. Base64 sert uniquement à transporter des données binaires dans des contextes limités au texte (URLs, JSON, emails), jamais à protéger une information.

Burp Suite & l'interception de requêtes

Proxy d'interception (Burp Suite)
Outil qui se positionne entre le navigateur et le serveur cible, permettant de visualiser et modifier chaque requête/réponse avant qu'elle ne poursuive son chemin. L'onglet Proxy → Intercept met en pause chaque échange HTTP, ce qui permet notamment d'examiner une redirection avant que le navigateur ne la suive automatiquement — révélant des informations normalement invisibles à l'œil nu.
curl
Outil en ligne de commande pour effectuer des requêtes HTTP/HTTPS. Contrairement à un navigateur, curl ne suit pas les redirections par défaut — il faut explicitement ajouter l'option -L pour cela. Sans cette option, curl affiche la réponse brute du serveur, headers inclus (avec -i), ce qui permet de voir directement le header Location sans jamais charger la page suivante.
Phase 02

Reconnaissance — La Page de Connexion

On accède à l'instance à l'adresse saturn.picoctf.net:59746, qui affiche un simple formulaire d'authentification, avec les identifiants de test déjà annoncés dans l'énoncé.

saturn.picoctf.net:59746 — Formulaire de connexion
"Help us test this form" — username:test et password:test pré-annoncés

On remplit le formulaire avec test / test! comme demandé, puis on soumet — dans un premier temps, de façon totalement classique, sans outil particulier.

Phase 03

L'Impasse du Navigateur

En soumettant le formulaire normalement, le navigateur suit silencieusement une ou plusieurs redirections en arrière-plan, puis affiche une page finale qui semble être un cul-de-sac :

"Welcome fellow Human — I was redirected here by a friend of mine but i couldnt find anything. Help me search for flags :-)"

Cette page /home propose même un champ de recherche "Search for flags" — un leurre qui invite à chercher activement, alors que la vraie information a déjà défilé sous nos yeux, cachée dans les redirections que le navigateur a suivies automatiquement sans les afficher.

Le navigateur a suivi la chaîne de redirections sans jamais montrer les headers Location intermédiaires — c'est exactement l'indice "any redirections?" du challenge

C'est ici que l'indice prend tout son sens : pour voir ce que le navigateur cache, il faut un outil capable d'intercepter chaque saut avant qu'il ne soit suivi.

Phase 04

Interception avec Burp Suite

On configure le navigateur pour passer par le proxy de Burp Suite, puis on active l'interception (Proxy → Intercept on) avant de soumettre à nouveau le formulaire. Cette fois, la requête POST /login est mise en pause et affichée dans Burp avant d'atteindre le serveur.

Burp Suite — Intercept — POST /login
Corps de la requête : username=test&password=test%21 (le "!" est encodé en %21)
URL encoding (%21)
Encodage qui remplace les caractères spéciaux non autorisés dans une URL par leur code hexadécimal précédé de %. Le point d'exclamation ! devient %21 — nécessaire car le mot de passe test! est transmis dans le corps de la requête au format application/x-www-form-urlencoded.

On transmet la requête (Forward) et on observe la réponse suivante, qui est elle-même une redirection interceptée.

Burp Suite — 1er saut de redirection intercepté
GET /next-page/id=bF90aGVfd2F5XzI1YmJhZTlhfQ== — un second paramètre id apparaît, encodé en Base64

Le premier saut redirige déjà vers une seconde URL contenant un paramètre id qui a toutes les caractéristiques d'un encodage Base64 (alphabet restreint, complété par ==). Burp propose un panneau Inspector qui détecte et décode automatiquement ce type de valeur sélectionnée.

Phase 05

Décodage Base64 des Fragments

En sélectionnant la valeur du paramètre id dans le panneau Inspector de Burp, celui-ci détecte automatiquement l'encodage Base64 et affiche le texte décodé en temps réel.

Burp Inspector — Décodage du 2e fragment
bF90aGVfd2F5XzI1YmJhZTlhfQ== décodé → "l_the_way_25bbae9a}"
Fragment 2 décodé : l_the_way_25bbae9a}

Ce fragment se termine par une accolade fermante } — signe caractéristique de la fin d'un flag au format picoCTF{...}. Il manque donc un premier fragment, qui doit correspondre au tout début du flag. On poursuit la chaîne de redirections dans Burp (bouton Forward) pour intercepter le saut précédent.

Burp Suite — 2e saut de redirection intercepté
GET /next-page/id=cGljb0NURntwcm94aWVzX2FsX3dheV8yNWJiYWU5YX0= — nouveau fragment Base64
Burp Inspector — Décodage du 1er fragment
Décodé → "picoCTF{proxies_al" — début du flag confirmé
Fragment 1 décodé : picoCTF{proxies_al

Les deux fragments sont désormais identifiés. Le nom du challenge et le contenu décodé convergent : "proxies all the way" — un clin d'œil au fait que le flag n'est révélé qu'en interceptant chaque étape de la chaîne de redirection avec un proxy, plutôt qu'en suivant passivement le parcours normal du navigateur.

Phase 06

Vérification Alternative avec curl

Pour confirmer la découverte indépendamment de Burp — et illustrer qu'un simple client en ligne de commande suffit — on reproduit la requête POST /login avec curl, en copiant fidèlement les en-têtes observés dans Burp.

Bash — Requête POST reproduite avec curl 
curl --path-as-is -i -s -k -X 'POST' \
  -H 'Host: saturn.picoctf.net:59746' \
  -H 'Content-Type: application/x-www-form-urlencoded' \
  --data-binary 'username=test&password=test%21' \
  'http://saturn.picoctf.net:59746/login'

L'option -i (inclure les headers) est essentielle ici : elle affiche la réponse complète du serveur, y compris le header Location, sans que curl ne suive automatiquement la redirection (comportement par défaut, contrairement à un navigateur).

Terminal — curl -i vers /login
HTTP/1.1 302 Found — Location: /next-page/id=cGljb0NURntwcm94aWVzX2FsX3dheV8yNWJiYWU5YX0=

Le résultat confirme exactement ce qu'affichait Burp : le premier fragment encodé apparaît directement dans le header Location de la réponse, visible en une seule commande terminal — sans navigateur, sans proxy graphique.

Bash — Décodage manuel en ligne de commande 
echo 'cGljb0NURntwcm94aWVzX2FsX3dheV8yNWJiYWU5YX0=' | base64 -d
Deux méthodes indépendantes (Burp Inspector et curl + base64 -d) confirment le même résultat
Phase 07

Assemblage du Flag

Il ne reste qu'à concaténer les deux fragments décodés dans leur ordre chronologique d'apparition dans la chaîne de redirection.

Fragment 1 picoCTF{proxies_al + Fragment 2 l_the_way_25bbae9a}
Assemblage manuel des deux fragments
picoCTF{proxies_al + l_the_way_25bbae9a} → picoCTF{proxies_all_the_way_25bbae9a}
Flag capturé
picoCTF{proxies_all_the_way_25bbae9a}
Flag validé et soumis avec succès sur la plateforme picoCTF
Phase 08

Recommandations de Sécurité

Ce challenge illustre un principe fondamental souvent négligé : l'encodage n'est pas une protection. Voici les enseignements à retenir pour éviter ce type de fuite dans une application réelle.

  • 01
    Ne jamais confondre encodage et chiffrement. Base64 est réversible instantanément par n'importe qui — il ne doit jamais servir à "cacher" une donnée sensible, seulement à transporter des données binaires dans un format texte.
  • 02
    Ne jamais transmettre de données sensibles dans une URL. Les paramètres d'URL (query string ou chemin) sont journalisés par les serveurs, les proxys, l'historique du navigateur et les logs — un canal peu sûr pour toute information confidentielle, même encodée.
  • 03
    Auditer systématiquement les chaînes de redirection. Un audit de sécurité web doit inclure l'inspection de chaque header Location d'une séquence de redirections — pas seulement la page finale visible.
  • 04
    Utiliser des identifiants opaques et non signifiants côté serveur. Un paramètre id transmis au client ne devrait jamais encoder directement une donnée métier — préférer un identifiant de session aléatoire résolu côté serveur.
  • 05
    Toujours tester avec un proxy d'interception avant mise en production. Des outils comme Burp Suite ou OWASP ZAP révèlent immédiatement ce type de fuite, invisible dans un test manuel classique au navigateur.
