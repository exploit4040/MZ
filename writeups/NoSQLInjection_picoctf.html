Présentation du Challenge

No Sql Injection est un challenge Web Exploitation de picoCTF 2024, créé par NGIRIMANA Schadrack, noté 200 points (difficulté moyenne). Particularité notable : le code source complet de l'application est fourni au format archive, ce qui transforme ce challenge en un exercice d'audit de code autant que d'exploitation pure.

L'énoncé est direct : "Can you try to get access to this website to get the flag?" — l'objectif est de contourner un mécanisme d'authentification. Le nom du challenge nous indique déjà la famille de vulnérabilité à explorer : une injection dans une base de données NoSQL, très probablement MongoDB étant donné son extrême popularité dans l'écosystème JavaScript/Node.js.

Phase 01

MongoDB — Architecture & Stockage des Données

Avant d'attaquer quoi que ce soit, il faut comprendre précisément ce qu'est MongoDB et comment il diffère fondamentalement d'une base de données relationnelle classique (MySQL, PostgreSQL) — cette différence est justement à l'origine de toute la classe de vulnérabilités que nous allons exploiter.

MongoDB
Système de gestion de base de données NoSQL orienté documents, open source, développé par MongoDB Inc. Contrairement aux bases relationnelles qui organisent les données en tables rigides avec des colonnes fixes, MongoDB stocke l'information sous forme de documents — des structures flexibles similaires à des objets JSON — regroupés en collections.

La hiérarchie de stockage MongoDB

MongoDB organise les données selon quatre niveaux hiérarchiques, du plus large au plus spécifique :

Niveau 1 — Déploiement
mongod (instance serveur) / Cluster Atlas
Le processus serveur qui héberge et gère l'ensemble des bases de données. Peut être une instance unique ou un cluster répliqué.
Niveau 2 — Database
ex: picoctf_app_db
Un conteneur logique regroupant plusieurs collections liées à une même application.
Niveau 3 — Collection
ex: users
L'équivalent d'une "table" en SQL, mais sans schéma fixe imposé par la base elle-même — chaque document peut théoriquement avoir des champs différents.
Niveau 4 — Document
{ email, password, token, ... }
L'équivalent d'une "ligne" en SQL — un objet structuré stocké au format BSON, identifié par un identifiant unique _id.

BSON : le format de stockage réel

BSON (Binary JSON)
Format binaire dans lequel MongoDB stocke physiquement chaque document sur le disque. BSON est une extension binaire de JSON : il conserve la même structure conceptuelle (clé/valeur, tableaux, objets imbriqués) mais ajoute des types de données supplémentaires non natifs à JSON — dates, entiers 64 bits, identifiants binaires (ObjectId) — et permet un parcours et une indexation beaucoup plus rapides qu'un texte JSON brut.
Concept SQL (relationnel)Équivalent MongoDB (NoSQL)
Base de donnéesDatabase
TableCollection
Ligne (row)Document (BSON)
ColonneChamp (field)
Clé primaire_id (ObjectId généré automatiquement)
Jointure (JOIN)Référence manuelle ou document imbriqué (embedding)
Schéma fixe (obligatoire)Schéma flexible, imposé uniquement côté application (ex: Mongoose)

C'est précisément ce dernier point — l'absence de schéma strict imposé par la base elle-même — qui rend possible la classe d'attaques que nous allons voir : MongoDB accepte volontiers des structures de requêtes complexes et imbriquées, y compris des opérateurs de comparaison, sans distinction stricte entre "donnée" et "instruction" si le code applicatif ne filtre pas correctement ce qu'il reçoit.

Requêtage MongoDB : le langage des opérateurs

MongoDB n'utilise pas de langage textuel comme SQL (SELECT * FROM users WHERE...). Ses requêtes sont elles-mêmes des documents JSON/BSON, où certaines clés spéciales préfixées par $ agissent comme des opérateurs de requête.

JavaScript — Exemple de requête MongoDB native 
// Recherche classique : trouver l'utilisateur avec cet email exact
db.users.findOne({ email: "bob@example.com" })

// Requête avec opérateur : trouver un utilisateur dont l'email N'EST PAS null
db.users.findOne({ email: { $ne: null } })

La seconde requête illustre le cœur du problème : la clé $ne ("not equal" — différent de) n'est pas une donnée, c'est une instruction de comparaison interprétée par le moteur MongoDB. Si une application transmet aveuglément une entrée utilisateur dans une requête sans la valider, un attaquant peut injecter ce type d'opérateur à la place d'une valeur attendue.

Où MongoDB est-il hébergé ? Les principaux services

MongoDB peut être déployé de multiples façons, du serveur auto-hébergé au service cloud entièrement managé. Voici les options les plus répandues dans l'industrie :

🍃
MongoDB Atlas
Le service cloud officiel de MongoDB Inc. — entièrement managé, disponible sur AWS, Azure et GCP. Standard de facto pour la production.
🟠
AWS DocumentDB
Service managé d'Amazon compatible avec l'API MongoDB, intégré nativement à l'écosystème AWS.
🔷
Azure Cosmos DB
Base de données multi-modèle de Microsoft, proposant une API compatible MongoDB parmi plusieurs autres.
🌊
DigitalOcean Managed MongoDB
Offre managée simplifiée, populaire auprès des petites équipes et startups pour son rapport simplicité/coût.
🐳
Auto-hébergement (Docker / VPS)
Déploiement manuel via conteneur Docker ou installation directe sur un serveur — contrôle total, mais responsabilité de sécurisation entièrement à la charge de l'équipe.
🧪
mongodb-memory-server
Bibliothèque Node.js qui démarre une instance MongoDB en mémoire, sans persistance disque — utilisée pour les tests automatisés et, comme ici, pour des environnements CTF éphémères.
Ce challenge utilise spécifiquement mongodb-memory-server — une base entièrement recréée en mémoire à chaque lancement d'instance, comme nous le confirmerons dans le code source

ODM (Object Document Mapper)
Bibliothèque qui fait le pont entre les objets JavaScript et les documents MongoDB, équivalent d'un ORM pour les bases SQL. Mongoose permet de définir des schémas (structure attendue, types, contraintes) au niveau applicatif — mais attention : ces schémas ne bloquent pas nécessairement les opérateurs de requête malveillants injectés dans les filtres de recherche (comme findOne()), seulement la structure des documents insérés.

L'information la plus précieuse ici est la confirmation que le flag est stocké directement dans la base de données, comme valeur du champ token d'un utilisateur pré-enregistré. L'objectif devient donc clair : trouver un moyen de lire ce document utilisateur sans connaître ses identifiants réels.

Phase 04

Théorie de l'Injection NoSQL

Avant de construire un payload, on approfondit la recherche théorique sur cette classe de vulnérabilité afin de comprendre exactement quel mécanisme exploiter.

recherchgoogle.png — Recherche MongoDB authentication bypass
Principe confirmé : l'opérateur $ne permet de contourner une vérification d'identifiants
NoSQL Injection
Classe de vulnérabilité analogue à l'injection SQL, mais ciblant les bases de données NoSQL. Elle survient quand une application transmet une entrée utilisateur non validée directement dans une requête, permettant à l'attaquant d'y injecter des opérateurs de requête plutôt que de simples valeurs littérales — modifiant ainsi la logique de la requête elle-même.

Anatomie de l'attaque par l'opérateur $ne

Une route de connexion vulnérable ressemble typiquement à ceci côté serveur :

JavaScript — Logique de connexion vulnérable (schéma général) 
const { email, password } = req.body;

// DANGER : email et password sont utilisés tels quels dans la requête,
// sans vérifier qu'il s'agit bien de chaînes de caractères simples
const user = await User.findOne({ email: email, password: password });

if (user) {
  // connexion réussie
}

Si l'application ne force pas email et password à être des chaînes de caractères strictes, un attaquant peut soumettre une structure au lieu d'une valeur :

JavaScript — Requête résultante après injection 
// Payload injecté : { "$ne": null } au lieu d'une chaîne classique
const user = await User.findOne({
  email: { $ne: null },
  password: { $ne: null }
});

// Traduction : "trouve le PREMIER utilisateur dont l'email n'est
// pas null ET dont le password n'est pas null" — c'est-à-dire
// n'importe quel utilisateur existant dans la collection !
$ne (Not Equal)
Opérateur retournant vrai pour tout document dont le champ est différent de la valeur donnée. { $ne: null } correspond à "n'importe quelle valeur non nulle" — donc quasiment tous les documents.
$gt / $lt (Greater/Less Than)
Opérateurs de comparaison numérique ou lexicographique, exploitables de façon similaire pour extraire des données caractère par caractère (technique dite "blind NoSQL injection").
$regex
Opérateur d'expression régulière — souvent exploité pour deviner un mot de passe progressivement en testant des préfixes de plus en plus longs.
findOne()
Méthode Mongoose/MongoDB retournant le premier document correspondant aux critères — dans une attaque $ne, elle retourne simplement le premier utilisateur de la collection, quel qu'il soit.
La faille ne vient jamais de MongoDB lui-même — le moteur exécute fidèlement la requête qu'on lui donne. La vulnérabilité réside entièrement dans le code applicatif qui fait confiance à l'entrée utilisateur sans validation de type

Phase 07

Recommandations de Sécurité

Cette vulnérabilité, bien que spécifique à MongoDB, illustre un principe universel en sécurité applicative : ne jamais faire confiance à une entrée utilisateur, quelle que soit la technologie de stockage utilisée en aval.

  • 01
    Valider strictement le type des entrées utilisateur. email et password doivent être forcés en chaînes de caractères (String(req.body.email)) avant toute utilisation dans une requête — cela neutralise immédiatement l'injection d'objets.
  • 02
    Utiliser une bibliothèque de sanitization dédiée. Le package express-mongo-sanitize supprime automatiquement toute clé commençant par $ ou contenant un point dans les objets entrants — une protection standard pour toute API Express/MongoDB.
  • 03
    Définir des schémas de validation stricts (Joi, Zod, Yup). Valider la forme exacte du corps de requête attendu avant qu'il n'atteigne la logique métier, en rejetant tout champ ou type inattendu.
  • 04
    Ne jamais stocker de secrets en clair, même encodés. Le Base64 n'est pas un chiffrement — le token exposé ici aurait dû être un identifiant de session opaque et signé (JWT avec signature vérifiée), jamais une donnée directement décodable.
  • 05
    Auditer le code source régulièrement, en particulier les points d'entrée utilisateur. Toute route recevant des données externes (formulaires, API) mérite une revue spécifique de la façon dont ces données sont utilisées dans les requêtes vers la base de données.
