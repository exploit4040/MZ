Phase 00

Présentation du Challenge

SQLiLite est un challenge Web Exploitation de picoCTF 2022, créé par Moubarak Mikail, noté 300 points (difficulté moyenne). Le nom du challenge est un jeu de mots explicite entre "SQL injection" et "SQLite" — le moteur de base de données ciblé ici.

challenge_card.png — picoCTF
SQLiLite — Web Exploitation, Moyen, 300 pts, picoCTF 2022. Indice : "admin est l'utilisateur que vous souhaitez connecter comme."

L'indice fourni est sans ambiguïté : l'objectif est de se connecter en tant que admin — sans en connaître le mot de passe. C'est la signature typique d'un exercice de contournement d'authentification par injection SQL.

Phase 01

SQLite — Architecture & Stockage des Données

Avant d'exploiter quoi que ce soit, comprenons précisément ce qu'est SQLite et en quoi son architecture diffère radicalement des bases de données "classiques" comme MySQL ou PostgreSQL — cette différence a une importance directe sur la façon dont une application web l'utilise, et donc sur sa surface d'attaque.

SQLite
Moteur de base de données relationnelle embarqué, open source, qui ne fonctionne pas comme un serveur séparé auquel on se connecte via le réseau. À la place, SQLite est une bibliothèque directement liée au programme qui l'utilise, et l'intégralité de la base de données — tables, index, données — est stockée dans un unique fichier sur le disque.

Concrètement, il n'existe aucun processus serveur à démarrer, aucun port réseau à ouvrir, aucune authentification réseau à configurer pour SQLite — l'application appelle directement les fonctions de la bibliothèque, qui lit et écrit dans le fichier .db comme n'importe quel autre fichier du système.

Structure interne du fichier SQLite

En interne, un fichier SQLite est organisé en pages de taille fixe (généralement 4096 octets), assemblées selon une structure d'arbre B-tree (arbre équilibré) pour chaque table et chaque index — cette structure permet des recherches, insertions et suppressions efficaces même sur des fichiers de plusieurs gigaoctets.

ComposantRôle
Page d'en-têteMétadonnées du fichier — version du format, taille de page, nombre de pages
sqlite_masterTable système cataloguant tous les objets de la base (tables, index, vues, triggers)
B-tree de tableStructure arborescente stockant les lignes d'une table, indexées par rowid
B-tree d'indexStructure arborescente accélérant les recherches sur une colonne indexée
Journal / WALFichier temporaire garantissant l'atomicité des transactions (mode rollback ou Write-Ahead Logging)

Où et pourquoi SQLite est-il utilisé en production ?

Contrairement à une idée reçue, SQLite n'est pas réservé aux petits projets — c'est probablement le moteur de base de données le plus déployé au monde en nombre d'instances actives, grâce à son intégration native dans d'innombrables systèmes :

Contexte d'usageExemple concret
Navigateurs webChrome et Firefox stockent historique, cookies et données de session dans des bases SQLite
Applications mobilesAndroid et iOS intègrent SQLite nativement pour le stockage local des applications
Systèmes embarqués / IoTObjets connectés à ressources limitées, sans besoin de serveur de base de données
Applications de bureauNombreux logiciels (gestionnaires de mots de passe, éditeurs) utilisent SQLite comme format de fichier
Prototypage & CTFSimplicité de déploiement — aucune configuration serveur nécessaire, cas d'usage de ce challenge
Important pour ce challenge : que la base soit SQLite, MySQL ou PostgreSQL, la logique de l'injection SQL classique reste quasiment identique — c'est la couche applicative qui construit la requête de façon non sécurisée, pas le moteur de base de données lui-même
Phase 02

Reconnaissance de l'Application

L'instance présente un formulaire de connexion simple, sans fonctionnalité visible au-delà des champs Username et Password.

saturn.picoctf.net:53891 — Formulaire de connexion
Formulaire "Log In" — champs Username et Password

On teste d'abord des identifiants classiques pour observer le comportement normal de l'application face à un échec de connexion.

Tentative — admin / password
"Login failed." — mais la page affiche la requête SQL exécutée en clair !

Cette réponse est extrêmement révélatrice — bien plus qu'un simple échec. L'application affiche la requête SQL brute construite côté serveur :

SQL — Requête exacte exposée par l'application 
SELECT * FROM users WHERE name='admin' AND password='password'
Fuite d'information critique : la structure exacte de la requête SQL est exposée au client — une erreur de configuration en elle-même (voir section sécurité)

Cette fuite nous donne exactement ce dont on a besoin pour construire une injection : on sait désormais que le champ Username est inséré directement entre guillemets simples dans la clause WHERE name='...', sans échappement apparent.

Phase 03

Théorie de l'Injection SQL

Injection SQL (SQLi)
Vulnérabilité survenant lorsqu'une application insère une entrée utilisateur non validée directement dans une requête SQL sous forme de texte concaténé, permettant à un attaquant de modifier la structure logique de la requête plutôt que de simplement fournir une valeur.

Anatomie de la requête vulnérable

Le code serveur construit probablement la requête ainsi, en concaténant directement l'entrée utilisateur dans la chaîne SQL :

Pseudo-code — Construction vulnérable de la requête 
const query = `SELECT * FROM users WHERE name='${username}' AND password='${password}'`;
db.execute(query);

Le problème est visible immédiatement : la variable username est insérée telle quelle entre les guillemets simples. Si cette variable contient elle-même un guillemet simple, elle "casse" la chaîne SQL prévue par le développeur — et tout ce qui suit devient du code SQL interprété, pas une simple donnée.

Le rôle du commentaire SQL (--)

Guillemet d'échappement (')
Termine prématurément la valeur de chaîne attendue par le développeur, permettant d'injecter du SQL arbitraire juste après.
Commentaire SQL (--)
Fait ignorer par le moteur SQL tout ce qui suit sur la ligne — utilisé pour neutraliser la vérification du mot de passe qui suivrait normalement dans la requête originale.
Clause WHERE
Filtre les lignes retournées par une requête SELECT selon une condition logique — c'est exactement cette condition que l'injection va manipuler.
Authentication bypass
Résultat final de l'attaque : obtenir un accès authentifié sans connaître le moindre identifiant valide, en trompant la logique de vérification elle-même.

En soumettant admin'-- comme nom d'utilisateur, la requête devient :

SQL — Requête après injection 
SELECT * FROM users WHERE name='admin'--' AND password='...'
                              ↑ tout ce qui suit -- est ignoré par SQLite

La requête effectivement exécutée se réduit à :

SQL — Requête réellement interprétée 
SELECT * FROM users WHERE name='admin'

La vérification du mot de passe a totalement disparu de la logique exécutée — la requête retourne l'utilisateur admin sans jamais valider aucun mot de passe.

admin'-- → Guillemet ferme la chaîne name → -- commente le reste → Clause password neutralisée
Phase 04

Exploitation — Contournement d'Authentification

On saisit le payload directement dans le champ Username, en laissant le champ Password vide — puisque, comme démontré, il ne sera jamais évalué.

Injection du payload
Username: admin'-- / Password: (laissé vide)
Payload — Champ Username 
admin'--

On soumet le formulaire. La réponse confirme immédiatement le succès du contournement.

Résultat — Authentification contournée
"Connecté! Mais pouvez-vous voir le drapeau, il est au clair." — requête confirmée sans clause password
SQL — Requête confirmée par l'application elle-même 
SQL query: SELECT * DEs utilisateurs WHERE name='admin'-' AND password=''
Authentification contournée avec succès — connecté en tant qu'admin sans mot de passe valide

Le message affiché — "pouvez-vous voir le drapeau, il est au clair" — est un indice délibéré du créateur : le flag n'est pas affiché visuellement dans la page rendue, mais il est présent quelque part dans le code, "en clair" (non chiffré, non filtré). C'est une invitation directe à inspecter le DOM de la page.

Phase 05

Extraction du Flag

Attribut HTML hidden
L'attribut hidden masque visuellement un élément à l'affichage (équivalent à display: none), mais son contenu reste parfaitement présent dans le code source et accessible via les outils de développement du navigateur. Masquer visuellement une donnée sensible n'équivaut jamais à la sécuriser.

On ouvre les DevTools du navigateur (onglet Inspecteur) pour examiner la structure complète du DOM généré par la page.

DevTools — Inspecteur DOM
Élément <p hidden=""> contenant "Your flag is: picoCTF{...}"
HTML — Élément caché contenant le flag 
<p hidden="">
    Your flag is: picoCTF{L00k5_l1k3_y0u_solv3d_it_d3c660ac}
</p>

Le flag était bien présent dans le HTML dès le chargement de la page — simplement masqué visuellement par CSS/attribut HTML, jamais réellement protégé côté serveur.

Phase 06

Aller Plus Loin — Techniques d'Injection SQL Avancées

Le contournement par commentaire ('--) n'est que la porte d'entrée du monde des injections SQL. Dans des scénarios réels — pentest, bug bounty, CTF plus avancés — voici les techniques essentielles à maîtriser pour aller au-delà d'un simple bypass d'authentification.

01
Injection UNION-based
Combine le résultat d'une requête légitime avec une requête arbitraire injectée, permettant d'extraire des données d'autres tables (mots de passe, emails, tokens).
' UNION SELECT username,password FROM users--
02
Blind SQLi — booléenne
Quand aucune donnée n'est affichée directement, on déduit des informations en observant si la page change de comportement selon une condition vraie ou fausse.
' AND (SELECT SUBSTR(password,1,1) FROM users)='a'--
03
Blind SQLi — temporelle
Variante où aucune différence visuelle n'existe : on mesure le délai de réponse du serveur pour déduire si une condition est vraie (via des fonctions de pause SQL).
'; SELECT CASE WHEN (1=1) THEN randomblob(100000000) END--
04
Error-based SQLi
Exploite les messages d'erreur détaillés renvoyés par la base pour extraire des données directement dans le texte de l'erreur — exactement le type de fuite observée en phase 02 de ce write-up.
' AND updatexml(1,concat(0x7e,(SELECT password FROM users LIMIT 1)),1)--
05
Stacked queries
Certains moteurs permettent d'exécuter plusieurs requêtes séparées par un point-virgule dans une seule injection — potentiellement destructeur (DROP, INSERT, UPDATE).
admin'; DROP TABLE users;--
06
Second-order injection
Le payload malveillant est stocké sans dommage immédiat (ex: lors d'une inscription), puis exécuté plus tard lorsqu'une autre fonctionnalité réutilise cette donnée dans une requête non sécurisée.
Stocké au signup, déclenché à la recherche

Méthodologie professionnelle pour identifier une injection

  1. Fuzzing des caractères spéciaux — tester ', ", --, #, ; sur chaque champ pour détecter des erreurs révélatrices.
  2. Observation différentielle — comparer le comportement de l'application entre une entrée valide, une entrée invalide, et une entrée cassant potentiellement la syntaxe.
  3. Confirmation logique — tester des conditions toujours vraies (' OR '1'='1) versus toujours fausses (' OR '1'='2) pour confirmer l'injection avant extraction.
  4. Extraction méthodique — une fois confirmée, utiliser UNION si possible, sinon basculer en mode blind (booléen ou temporel).
  5. Automatisation — des outils comme sqlmap automatisent l'ensemble de ce processus une fois le point d'injection identifié manuellement.
Ces techniques ne doivent être pratiquées que sur des environnements légaux : plateformes CTF, labs dédiés (PortSwigger Academy, HackTheBox), ou avec autorisation écrite explicite
Phase 07

Recommandations de Sécurité

Ce challenge cumule en réalité deux vulnérabilités distinctes : l'injection SQL elle-même, et une fuite d'information par messages d'erreur trop verbeux. Voici les corrections pour chacune.

  • 01
    Utiliser systématiquement des requêtes préparées (prepared statements). Les paramètres doivent être liés séparément du texte de la requête via des placeholders (? ou :param), empêchant structurellement toute donnée utilisateur d'altérer la syntaxe SQL.
  • 02
    Ne jamais exposer les requêtes SQL brutes ou les messages d'erreur détaillés au client. Ce challenge montre précisément pourquoi : la requête affichée en clair a directement renseigné l'attaquant sur la structure exacte à casser.
  • 03
    Appliquer le principe du moindre privilège pour le compte de base de données. Le compte utilisé par l'application web ne devrait avoir accès qu'aux tables et opérations strictement nécessaires — jamais de droits d'administration complets.
  • 04
    Ne jamais masquer une donnée sensible uniquement côté client (CSS/HTML). L'attribut hidden ou display:none ne constitue en aucun cas une protection — toute donnée sensible ne doit tout simplement pas être envoyée au navigateur si l'utilisateur n'y a pas droit.
  • 05
    Déployer un WAF (Web Application Firewall) en défense complémentaire. Une couche de détection de patterns d'injection SQL ajoute une protection supplémentaire, sans jamais remplacer la correction du code applicatif lui-même.

Exemple de correction avec requête préparée

JavaScript — Version sécurisée avec requête préparée 
// Le paramètre est lié séparément — impossible d'altérer la syntaxe SQL
const stmt = db.prepare("SELECT * FROM users WHERE name = ? AND password = ?");
const user = stmt.get(username, password);
